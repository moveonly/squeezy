#### `freeform-answer-cursor-pinned-to-end` — Freeform plan-mode answer renders its caret at the end regardless of where you edit

- **Severity:** high · **Category:** bug · **Risk:** low · **Effort:** small · **Disposition:** ✅ fix
- **Files:** `crates/squeezy-tui/src/lib.rs:25254`, `crates/squeezy-tui/src/lib.rs:25261`, `crates/squeezy-tui/src/input.rs:1493`, `crates/squeezy-tui/src/input.rs:1521`
- **Problem:** handle_request_user_input_key fully supports mid-text editing of the freeform answer: Left/Right move answer_cursor (input.rs:1493-1504), Backspace/Delete/insert all splice at answer_cursor (input.rs:1521-1551). But the renderer ignores answer_cursor entirely: format_request_user_input_menu_lines takes only `input: &str` and always draws `compact_text(input, 200)` followed by a `▌` block at the END (lib.rs:25261-25262). The code comment even admits it ('the cursor sits at answer_cursor bytes, which we don't have here — approximate'). Result: a user who arrows back into the middle of their answer to fix a typo sees the caret stuck at the end while characters insert/delete somewhere invisible — the editing keys appear broken. It also truncates display at 200 chars so on a long answer the block detaches from the real edit point entirely.
- **Fix:** Thread the byte cursor into the renderer. Change `fn format_request_user_input_menu_lines(request, selected, input: &str)` to also take `answer_cursor: usize`, and at both call sites (lib.rs:33093-33097 and 33149-33153) pass `pending.answer_cursor`. In the else branch (lib.rs:25254-25263), clamp the cursor to a char boundary, split the answer into `head = &input[..cursor]` / `tail = &input[cursor..]`, push `head` styled with entry_style, push the `▌` cursor block, then push `tail` styled with entry_style (mirroring how the composer paints its inline caret). Keep the empty-input branch as-is.
- **Verifier-refined fix:** Direction is correct (thread answer_cursor into the renderer and paint head/cursor/tail), but two details the proposal omits: (1) clamp_byte_cursor is a private fn in input.rs (lib.rs:120 `mod input;`), not callable from lib.rs — either make it pub(crate) or inline the clamp in lib.rs (cursor.min(input.len()) then walk back to a char boundary). (2) The current code truncates via compact_text(input, 200); splitting raw input at a cursor > 200 would place the caret beyond the shown text. Replace the single compact_text call with: clamp cursor to a char boundary; head = compact_text(&input[..cursor], 200); tail = &input[cursor..] (also compact-limited so the line stays bounded, e.g. compact_text(tail, 200)); push Span::styled(head, entry_style), Span::styled("▌", cursor_style), Span::styled(tail, entry_style). Keep the empty-input branch unchanged. Note compact_text only replaces \n with spaces (1:1 byte-wise) so cursor byte offsets remain valid for slicing the raw input before compacting each half.
- **Verdict:** real=True · safe_fix=True · confidence=high — Confirmed in current code. Input handling fully supports mid-text editing: input.rs:1493-1504 move answer_cursor on Left/Right, and insert_answer_char/delete_answer_* (1521-1551) all splice at clamp_byte_cursor(answer_cursor). But the renderer's signature is fn format_request_user_input_menu_lines(request, selected, input: &str) (lib.rs:25163-25167) with no cursor, and the else branch always draws `compact_text(input, 200)` then a `▌` block at the END (lib.rs:25261-25262), with an in-code comment admitting "the cursor sits at answer_cursor bytes, which we don't have here — approximate". Both call sites (lib.rs:33093 and 33149) pass only &pending.answer though pending.answer_cursor is in scope. So arrowing into the middle leaves the visible caret pinned at the end — real high-severity UX bug.

#### `vertical-cursor-byte-column` — Up/Down line movement uses a byte column, mis-placing the caret across multi-byte (CJK/emoji) lines

- **Severity:** medium · **Category:** bug · **Risk:** medium · **Effort:** medium · **Disposition:** ✅ fix
- **Files:** `crates/squeezy-tui/src/input.rs:1037`, `crates/squeezy-tui/src/input.rs:1044`, `crates/squeezy-tui/src/input.rs:1052`, `crates/squeezy-tui/src/input.rs:1065`
- **Problem:** move_input_cursor_up/down compute the column as a raw byte offset `col = cursor - curr_start` and re-apply it as a byte offset on the target line: `prev_start + col.min(prev_len)` (input.rs:1044) and `next_start + col.min(next_len)` (input.rs:1065). When the source or target line contains multi-byte characters, this byte column does not correspond to the same visual column, and the resulting offset can land mid-glyph. The render path's input_cursor() then snaps it down to the nearest char boundary, so the caret jumps to an unexpected position (typically left of where the user expects) when arrowing vertically through CJK/emoji/accented text. ASCII text is unaffected, which is why it slips through.
- **Fix:** Track the column in characters, not bytes. In move_input_cursor_up: let `col = app.input[curr_start..cursor].chars().count();` then set the target by taking that many chars of the previous line: `let prev = &app.input[prev_start..prev_end]; app.input_cursor = prev_start + prev.char_indices().nth(col).map(|(i,_)| i).unwrap_or(prev.len());`. Apply the symmetric change in move_input_cursor_down over `next_start..next_end`. This keeps the caret on the same character column and always on a boundary.
- **Verifier-refined fix:** Track the column in characters. In move_input_cursor_up, replace line 1037 with `let col = app.input[curr_start..cursor].chars().count();` and line 1044 with `let prev = &app.input[prev_start..prev_end]; app.input_cursor = prev.char_indices().nth(col).map(|(i, _)| prev_start + i).unwrap_or(prev_end);` (use prev_end, which is the byte end of the previous line, as the clamp). Symmetrically in move_input_cursor_down: replace line 1053 with `let col = app.input[curr_start..cursor].chars().count();` and line 1065 with `let next = &app.input[next_start..next_end]; app.input_cursor = next.char_indices().nth(col).map(|(i, _)| next_start + i).unwrap_or(next_end);`. This keeps the same character column and always lands on a boundary; prev_len/next_len become unused and can be removed.
- **Verdict:** real=True · safe_fix=True · confidence=high — Confirmed verbatim: input.rs:1037 `let col = cursor - curr_start;` (a raw byte delta) is re-applied as a byte offset on the target line at :1044 `app.input_cursor = prev_start + col.min(prev_len);` and :1065 `app.input_cursor = next_start + col.min(next_len);`. `app.input_cursor` is a byte offset, and `text_cursor` (input.rs:871-877) snaps a non-boundary offset DOWN via `while cursor>0 && !is_char_boundary { cursor-=1 }`, so a byte column landing mid-glyph on a CJK/emoji target line gets pushed left — exactly the described mis-placement. ASCII is unaffected since byte==char column there. No existing test covers vertical movement, so it slips through.
