# Website Feature Research

This note inventories implemented user-facing convenience and product features
that can be referenced on the website. Keep the positioning modest: these
features make Squeezy safer and easier to operate around the token-saving
core; they are not the core value proposition by themselves.

Use public-safe wording only. Avoid promising a hard security boundary,
marketplace-style extension support, or autonomous cloud-agent behavior unless
the implementation changes first.

## Positioning Summary

Public framing:

- Squeezy is a cost-aware coding agent that prefers local semantic navigation
  before spending model tokens.
- The surrounding product features are usability and safety controls for real
  coding sessions: permission prompts, resumable sessions, local skills,
  controlled web/MCP access, and TUI workflows.
- Prefer phrases such as "helps keep actions reviewable", "asks before network
  and external-tool access by default", and "local filesystem skills" over
  stronger claims such as "secure sandbox", "plugin marketplace", or
  "autonomous multi-agent system".

## Website Copy Guardrails

Use:

- "guardrails", "reviewable", "permissioned", "local", "resumable",
  "bounded", "controlled", "opt-in".
- "semantic navigation first" and "token-saving core" as the center of gravity.
- "web/MCP access is available when configured and approved" rather than
  "browses the web" or "connects to any tool".
- "local filesystem skills" rather than "plugins" unless the copy explains
  that they are not marketplace extensions.

Avoid:

- "secure sandbox" without platform caveats.
- "autonomous multi-agent execution" for subagents.
- "guaranteed rollback" for checkpoints.
- "billing-accurate cost tracking"; use "estimated" or "provider-reported".
- "marketplace", "remote extensions", or "plugin runtime" for skills.

## Good Public Phrases

- "Spend fewer model tokens by navigating code locally first, then use
  permissioned tools when work needs edits, shell commands, web evidence, or
  external MCP servers."
- "Review file edits, shell commands, web fetches, and MCP calls before they
  run, with optional session rules for trusted repeated actions."
- "Resume, label, fork, and export local sessions so longer coding work remains
  inspectable."
- "Use local skills for reusable project instructions without turning them into
  a remote plugin system."
- "Route narrow mechanical turns to cheaper models, and fall back to the parent
  model when the task becomes ambiguous or tool-heavy."
